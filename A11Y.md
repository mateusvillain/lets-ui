# Let's UI — Accessibility Specification

This document defines the accessibility contract for the Let's UI design system. All components must conform to **WCAG 2.1 Level AA**. Conformance is verified through the Storybook A11y addon, manual keyboard testing, and screen reader spot-checks.

---

## Standards target

| Standard     | Level | Scope                            |
| ------------ | ----- | -------------------------------- |
| WCAG 2.1     | AA    | All components and foundations   |
| HTML spec    | —     | Semantic elements and ARIA usage |
| WAI-ARIA 1.2 | —     | Role, property, and state usage  |

---

## Universal requirements

Every component, regardless of type, must satisfy these baseline rules.

### Color and contrast

- Text on background: minimum **4.5 : 1** contrast ratio.
- Large text (≥ 18 pt / 14 pt bold) and UI components: minimum **3 : 1**.
- Status never conveyed by color alone — always pair with icon, label, or pattern.
- All token pairs (text on background, icon on surface) must be verified when adding or modifying tokens.

### Focus visibility

- All interactive elements must have a visible focus indicator.
- Use the `focus-ring($variant)` utility mixin on `:focus-visible`. Never target `:focus` alone.
- Focus rings must meet **3 : 1** contrast against adjacent colors.
- Do not suppress the browser default outline without replacing it with an equivalent indicator.

### Disabled state

Form controls and buttons that can be disabled must expose the state through **both** mechanisms:

```html
<button disabled aria-disabled="true">…</button>
```

The HTML `disabled` attribute removes the element from focus order; `aria-disabled="true"` keeps it in the DOM announcement. Use both. Disabled elements must be skipped in programmatic keyboard navigation loops.

### Decorative icons and images

All SVGs that are decorative (icon next to a label, status icon, spinner) must be hidden from the accessibility tree:

```html
<svg aria-hidden="true" focusable="false">…</svg>
```

Images that carry meaning need `alt` text. Images used as decoration use `alt=""`.

### Motion and animation

Respect `prefers-reduced-motion`. Transitions and animations must be suppressed or significantly reduced when the OS preference is set.

## ARIA usage rules

- Only use a role when the native HTML element does not already convey it. `<button>` never needs `role="button"`.
- `aria-label` is for elements with no visible text label (icon-only buttons, landmark regions without headings).
- `aria-labelledby` links to an existing visible element by ID. Prefer it over `aria-label` when the label is already in the DOM.
- `aria-describedby` links to supplementary text (hints, error messages). It does not replace a label.
- Never use `aria-hidden="true"` on an element that contains focusable children.
- `aria-disabled="true"` does not prevent keyboard focus or event firing — the component must handle that in JavaScript.

## Form association

All form controls use the modern `ElementInternals` API. This applies to: `lui-button`, `lui-checkbox`, `lui-input`, `lui-radio`, `lui-radio-group`, `lui-select`, `lui-switch`, `lui-textarea`.

### Required implementation

```ts
static formAssociated = true;
private _internals: ElementInternals;

constructor() {
  super();
  this._internals = this.attachInternals();
}

formDisabledCallback(disabled: boolean) {
  this.disabled = disabled;
}

formResetCallback() {
  this.value = this.defaultValue; // restore to initial attribute value
  this._internals.setFormValue(null);
}
```

### Validity

Set validity state explicitly via `_internals.setValidity()`. Do not rely on the native element's own validity when the component wraps it inside shadow DOM.

```ts
if (this.required && !this.value) {
  this._internals.setValidity({ valueMissing: true }, 'This field is required');
} else {
  this._internals.setValidity({});
}
```

### Error exposure

- Bind the error message element's ID to `aria-describedby` on the control.
- Set `aria-invalid="true"` when `error` is true (not yet implemented uniformly — add when touching a form component).
- The error message must be visible in the DOM (not toggled with `display: none`) for `aria-describedby` to work.

## Focus management

### Overlay components (Modal, Drawer)

When an overlay opens:

1. Move focus into the overlay — call `requestAnimationFrame(() => el.focus())` on the dialog or the first focusable element.
2. Trap Tab/Shift+Tab within the overlay using a focus-trap loop:

   ```ts
   const focusable = el.querySelectorAll(
     'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
   );
   ```

3. When the overlay closes, restore focus to the element that triggered it.
4. Apply the `inert` attribute to the overlay element when closed; remove it on open.

### Keyboard navigation components (Tabs, Dropdown Menu)

Use **roving tabindex**:

- The active item has `tabindex="0"`. All others have `tabindex="-1"`.
- Arrow keys move the active item and call `.focus()` on it.
- After focus moves, update `tabindex` values.
- Tab key leaves the widget entirely — do not intercept it inside the navigation loop.

## Keyboard maps

### Button / Icon Button

| Key     | Action   |
| ------- | -------- |
| `Enter` | Activate |
| `Space` | Activate |

Icon Button requires a non-empty `aria-label`. There is no exception.

### Modal / Drawer

| Key         | Action                                                      |
| ----------- | ----------------------------------------------------------- |
| `Escape`    | Close overlay; restore focus to trigger                     |
| `Tab`       | Move to next focusable element (trapped inside overlay)     |
| `Shift+Tab` | Move to previous focusable element (trapped inside overlay) |

Required ARIA on the dialog element:

```html
<div
  role="dialog"
  aria-modal="true"
  aria-labelledby="{id}-title"
  aria-describedby="{id}-body"
></div>
```

Required ARIA on the trigger:

```html
<button
  aria-haspopup="dialog"
  aria-controls="{id}-dialog"
  aria-expanded="false|true"
></button>
```

### Tabs

| Key          | Action                        |
| ------------ | ----------------------------- |
| `ArrowLeft`  | Activate previous enabled tab |
| `ArrowRight` | Activate next enabled tab     |
| `Home`       | Activate first enabled tab    |
| `End`        | Activate last enabled tab     |

All arrow keys must call `event.preventDefault()`. Disabled tabs are skipped. Required ARIA:

```html
<div role="tablist">
  <button
    role="tab"
    aria-selected="true|false"
    aria-controls="{id}-panel-N"
    tabindex="0|-1"
  >
    <div role="tabpanel" aria-labelledby="{id}-tab-N"></div>
  </button>
</div>
```

### Dropdown Menu

| Key               | Action                                     |
| ----------------- | ------------------------------------------ |
| `ArrowDown`       | Move to next item (circular)               |
| `ArrowUp`         | Move to previous item (circular)           |
| `ArrowRight`      | Open submenu; focus first submenu item     |
| `ArrowLeft`       | Close submenu; return focus to parent item |
| `Escape`          | Close menu; restore focus to trigger       |
| `Tab`             | Close menu                                 |
| `Enter` / `Space` | Activate item                              |

Disabled items are skipped in navigation. Required ARIA:

```html
<ul role="menu">
  <li role="presentation">
    <button role="menuitem" aria-disabled="true|false">
      <button role="menuitem" aria-haspopup="menu" aria-expanded="true|false">
        <!-- submenu parent -->
        <li role="separator">
          <!-- Trigger -->
          <button
            aria-haspopup="menu"
            aria-expanded="true|false"
            aria-controls="{id}-menu"
          ></button>
        </li>
      </button>
    </button>
  </li>
</ul>
```

### Radio Group

| Key                        | Action                 |
| -------------------------- | ---------------------- |
| `ArrowDown` / `ArrowRight` | Move to next radio     |
| `ArrowUp` / `ArrowLeft`    | Move to previous radio |
| `Space`                    | Select focused radio   |

The group uses a `<fieldset>` + `<legend>` structure. Each radio is a native `<input type="radio">`. The group element is `lui-radio-group`; keyboard handling is delegated to the browser through native inputs within the same `name` group.

### Select / NativeSelect

Delegates keyboard handling entirely to the native `<select>` element. Do not override or suppress native browser behavior.

### Input (number type)

The increment/decrement step buttons:

| Key         | Action          |
| ----------- | --------------- |
| `ArrowUp`   | Increment value |
| `ArrowDown` | Decrement value |

Step buttons require an `aria-label` describing their action (e.g., `"Aumentar"` / `"Diminuir"`).

### Tooltip

- The trigger element must be focusable. If the slotted trigger contains no focusable element, add `tabindex="0"` to the wrapper.
- Set `aria-describedby` on the trigger pointing to the tooltip content.
- The tooltip content element has `role="tooltip"`.
- Show on `:focus-visible` as well as on hover.

### Alert

The `aria-live` value must match the urgency of the variant:

| Variant             | `aria-live` | Reason                                                                        |
| ------------------- | ----------- | ----------------------------------------------------------------------------- |
| `danger`, `caution` | `assertive` | Requires immediate attention; interrupts the screen reader                    |
| `success`, `info`   | `polite`    | Informational; waits for the screen reader to finish its current announcement |

```html
<!-- danger / caution -->
<div
  role="alert"
  aria-live="assertive"
  aria-atomic="true"
  aria-labelledby="{id}-title"
  aria-describedby="{id}-content"
>
  <!-- success / info -->
  <div
    role="alert"
    aria-live="polite"
    aria-atomic="true"
    aria-labelledby="{id}-title"
    aria-describedby="{id}-content"
  ></div>
</div>
```

## Component-specific contracts

### lui-button

| Property          | Requirement                                                    |
| ----------------- | -------------------------------------------------------------- |
| `aria-label`      | Required when the button has no visible text (icon-only usage) |
| `aria-disabled`   | Must match `disabled` attribute                                |
| `aria-busy`       | Must be `"true"` when `loading` is true                        |
| Native `<button>` | Always; never use `<div role="button">`                        |

### lui-icon-button

`aria-label` is **mandatory** — there is no visible text label. A missing or empty `aria-label` is a hard failure.

### lui-input

| Property           | Requirement                                       |
| ------------------ | ------------------------------------------------- |
| `<label for="…">`  | Always present; tied to the input via matching ID |
| `aria-describedby` | Points to hint and/or error message element       |
| `aria-disabled`    | Must match `disabled` attribute                   |
| `aria-invalid`     | Must be `"true"` when `error` is true             |

Password toggle button requires `aria-label` describing current action (`"Mostrar senha"` / `"Ocultar senha"`).

### lui-checkbox / lui-radio / lui-switch

| Property             | Requirement                                                                         |
| -------------------- | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| Native input element | `<input type="checkbox                                                              | radio">`always;`<input type="checkbox" role="switch">` for switch |
| `aria-checked`       | Explicitly set on switch (`role="switch"` does not inherit implicit state reliably) |
| `aria-label`         | Required when no visible `<label>` is present                                       |
| `aria-disabled`      | Must match `disabled` attribute                                                     |

### lui-radio-group

- Uses `<fieldset>` + `<legend>` — do not replace with generic `<div role="group">`.
- Error message connected via `aria-describedby` on the fieldset.
- Propagates `disabled` to all child `lui-radio` elements.

### lui-select / lui-native-select

- Native `<select>` element always used.
- Placeholder `<option>` has `disabled` and `hidden` attributes — it is not a valid choice.
- Arrow icon hidden: `aria-hidden="true"`.

### lui-modal / lui-drawer

- `inert` attribute applied when closed; removed when open.
- Close button aria-label must be descriptive: `"Fechar modal"` / `"Fechar drawer"`.
- SVG inside close button: `aria-hidden="true"`.
- `aria-hidden` on backdrop element (opposite value to `open`).

### lui-tabs

- Roving tabindex: active tab has `tabindex="0"`, others have `tabindex="-1"`.
- Disabled tabs require both `disabled` and `aria-disabled="true"`.
- Panel hidden state managed via the `hidden` attribute on `role="tabpanel"`.

### lui-dropdown-menu

- Trigger button always has `aria-haspopup="menu"` and `aria-expanded`.
- Submenu parent items have `aria-haspopup="menu"` and `aria-expanded`.
- `<li>` wrappers use `role="presentation"` to remove implicit list semantics.
- Dividers use `role="separator"`.

### lui-alert

- `role="alert"` + `aria-atomic="true"` always present.
- `aria-live` is variant-dependent: `"assertive"` for `danger` and `caution`; `"polite"` for `success` and `info`.
- Variant icons are decorative: `aria-hidden="true"`.
- Title and content IDs set for `aria-labelledby` and `aria-describedby`.

### lui-tooltip

- `role="tooltip"` on content element.
- Trigger must receive `aria-describedby` pointing to tooltip ID.
- If slotted trigger has no focusable element, the wrapper gets `tabindex="0"`.

## Testing

### Storybook A11y addon

The `@storybook/addon-a11y` addon is installed and configured. It runs axe-core against each story automatically.

The current config in `.storybook/preview.js` sets `test: 'todo'` — this must be changed to `test: 'error'` as stories mature, so that axe violations fail the test run. Do not ship a new component story without first passing the axe scan.

Enabling automated a11y tests per story:

```js
export default {
  parameters: {
    a11y: {
      config: {
        rules: [{ id: 'color-contrast', enabled: true }],
      },
    },
  },
};
```

### Manual keyboard testing checklist

Run this for every interactive component before marking it done:

- [ ] All interactive elements reachable by Tab alone.
- [ ] Visible focus indicator present at every stop.
- [ ] Correct action on Enter and/or Space.
- [ ] Arrow key navigation works where specified (tabs, menu, radio).
- [ ] Escape closes overlays and restores focus to the trigger.
- [ ] Disabled elements cannot be activated by keyboard.
- [ ] Focus does not escape modal/drawer while open.
- [ ] Focus returns correctly when overlay closes.

### Screen reader spot-check (VoiceOver / macOS)

- Open VoiceOver (`Cmd+F5`).
- Navigate to the component with Tab.
- Verify the announced role, name, and state match what is expected.
- Activate the component and verify state changes are announced.
- For alerts: verify the announcement fires on insertion.

Screen reader testing is not required for every PR, but must be done for new components and for any change to ARIA attributes, roles, or focus management.

- [ ] WCAG 2.1 AA contrast verified for all token pairs used.
- [ ] No information conveyed by color alone.
- [ ] All interactive elements keyboard-reachable.
- [ ] Correct ARIA role (implicit via native element preferred).
- [ ] `aria-label` or `aria-labelledby` present when no visible label exists.
- [ ] `aria-describedby` connects hints and error messages.
- [ ] `aria-disabled="true"` mirrors `disabled` attribute.
- [ ] `aria-invalid="true"` set when error state is active.
- [ ] Focus ring implemented with `focus-ring()` mixin on `:focus-visible`.
- [ ] Decorative SVGs marked `aria-hidden="true" focusable="false"`.
- [ ] Keyboard map implemented and verified (see component type above).
- [ ] `formAssociated` + `ElementInternals` for form controls.
- [ ] Story passes Storybook A11y addon scan with no violations.
- [ ] Manual keyboard test checklist completed.
